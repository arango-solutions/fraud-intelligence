This Product Requirements Document (PRD) defines the **Visualization & User Experience (UX)** subsystem for the Fraud Intelligence Demo. This layer serves as the "Narrative Interface," translating complex graph algorithms and AI reasoning into three distinct visual stories tailored for specific banking stakeholders.

## 1. Executive Summary

The UX subsystem is not merely a dashboard; it is a storytelling tool. It must demonstrate the **"Before & After"** impact of the platform.

* **Before:** Noisy data, disconnected entities, reactive rules.
* **After:** Resolved identities, visible fraud rings, predictive risk heatmaps.

The interface will be built as a unified **Web Application** (likely Streamlit) with three tabs corresponding to the three "Lenses" defined below.

---

## 2. Lens 1: The "Investigator" View (Graph Explorer)

**Target Persona:** Field Investigation Officer (FIO) at the bank.
**Core Question:** "Is this specific person (customer) part of a fraud ring?"

### 2.1 Feature: Interactive Network Graph

* **Requirement:** Render a force-directed graph of the specific sub-network surrounding a suspect entity.
* **Visual Elements:**
* **Nodes:** Color-coded by Type (`Person`=Blue, `BankAccount`=Green, `RealProperty`=Orange, `WatchlistEntity`=Red). Sized by `risk_score`.
* **Edges:** Directed arrows showing money flow (`transferred_to`) or relationships (`related_to`).


* **Interaction:** Pan, zoom, drag nodes to untangle clusters.

### 2.2 Feature: The "Entity Resolution Toggle" (The "Wow" Moment)

* **Goal:** Visually demonstrate how the platform defeats "Benami" proxies.
* **Mechanism:** A toggle switch labeled **"Apply Identity Resolution"**.
* **State A (Off):** Show the "Raw" graph.
* *Visual:* 3 separate nodes: "Rajesh Kumar", "R. Kumar", "Rajesh K." connected to different transactions. The fraud ring looks broken/disconnected.


* **State B (On):** Show the "Resolved" graph.
* *Visual:* The 3 nodes animate and collapse into a single **`GoldenRecord`** node. The edges merge, suddenly revealing a closed loop of circular trading that was previously invisible.


* **Data Source:** Subsystem 2 (`arango-entity-resolution`) output.

### 2.3 Feature: GraphRAG Explainer Sidebar

* **Goal:** Provide evidence for the "Why".
* **Mechanism:** Clicking any node opens a "Case File" sidebar.
* **Content:**
* **Entity Details:** Name, PAN, Risk Score.
* **The Evidence Path:** A visual lineage showing why the node is flagged.
* *Display:* "Linked to Watchlist ID #123 (2 hops away)."


* **Source Citations:** Display snippets of unstructured text linked via GraphRAG.
* *Display:* "Context found in **'Mumbai_Registrar_Deeds_2025.pdf'**: '...property sold to **Rajesh Kumar** for ₹50L (Circle Rate: ₹90L)...'".

## 3. Lens 2: The "Analyst" View (Interactive Reports)

**Target Persona:** Senior Analyst / Model Validator.
**Core Question:** "How did the AI reach this conclusion, and is it statistically significant?"

### 3.1 Feature: Automated Intelligence Reports

* **Requirement:** Render the HTML reports generated by the **Agentic Workflow** directly in the UI.
* **Content:** Executive summary, confidence scores, and algorithm parameters used (e.g., "PageRank Damping: 0.85").

### 3.2 Feature: Interactive Plotly Charts

The view must embed interactive charts that validate the fraud findings using statistical outliers.

#### **Chart A: The "Mule Hub" Detector (PageRank)**

* **Type:** Bar Chart (Sorted Descending).
* **X-Axis:** BankAccount IDs.
* **Y-Axis:** Influence Score (PageRank).
* **Visual Story:**
* Show top 20 accounts.
* Highlight the **"Hawala Broker"** bar in **Red**.
* *Annotation:* "BankAccount X has 50x higher influence than average retail users despite zero salary credits."


* **Tech:** Plotly JSON export from Subsystem 3.

#### **Chart B: The "Fraud Ring" Isolator (WCC)**

* **Type:** Histogram (Log Scale).
* **X-Axis:** Community Size (Number of Nodes).
* **Y-Axis:** Count of Communities.
* **Visual Story:**
* Most communities are size 2-5 (Families).
* One distinct bar exists at size **50+**.
* *Annotation:* "Anomalous cluster detected: 52 nodes connected by device ID (Potential Mule Ring)."


* **Tech:** Plotly Histogram generated by `ReportingSpecialist` agent.

## 4. Lens 3: The "Executive" View (Risk Heatmap)

**Target Persona:** Chief Risk Officer (CRO) / Compliance Head.
**Core Question:** "What is our systemic exposure to Real Estate fraud?"

### 4.1 Feature: Geospatial Risk Overlay

* **Requirement:** A map of the target region (e.g., Mumbai Metro) overlaid with risk data.
* **Visual Elements:**
* **Base Map:** Street map of India/Mumbai.
* **Heat Layer:** Aggregated `risk_score` from Subsystem 4.
* **Granularity:** Hex-bin or District polygons (e.g., "Andheri East", "Bandra Kurla Complex").


* **Data Source:** `RealProperty` and `Address` nodes with `lat`/`long` attributes generated by the Data Fabric.

### 4.2 Feature: The "Visual Alert"

* **Requirement:** Dynamic text overlay that summarizes the heatmap.
* **Logic:**
* *Input:* `docs/sentries_risk_heatmap.json` (from Subsystem 4).
* *Display:* "⚠️ **CRITICAL ALERT:** High concentration of undervalued property transactions detected in **Thane West** district."


* **Interaction:** Clicking the "Red" district filters the **Investigator View** (Lens 1) to only show entities in that location.

## 5. Technical Implementation Plan

### 5.1 Technology Stack

* **Frontend Framework:** **Streamlit**.
* *Why:* Fast iteration, native support for Python data structures, `st.plotly_chart`, and `pydeck` (maps).


* **Graph Visualization:** **`streamlit-agraph`** or a custom React component using `react-force-graph` embedded via `st.components.v1.html`.
* **Mapping:** **`pydeck`** (Deck.gl wrapper for Python). High-performance rendering of 10k+ points.

### 5.2 Data Integration

* **AQL Connectivity:** The UI will connect directly to ArangoDB via `python-arango` to fetch live graph data.
* **Agent Integration:** The UI will allow users to input text prompts which are sent to the `AgenticWorkflowRunner` API. The resulting HTML/JSON report is rendered back to the user.

### 5.3 UX Workflow (The Demo Script)

1. **Start (Executive View):** Show the map. "We see a risk spike in Mumbai."
2. **Click District:** Transition to **Analyst View**. "Let's analyze the transaction patterns there."
3. **Run Agent:** "Agent, find circular trading in this district." -> Agent runs -> Displays **Analyst Charts** (PageRank/WCC).
4. **Drill Down:** Select the top outlier from the chart. Transition to **Investigator View**.
5. **Toggle ER:** Show the disjointed nodes -> Toggle ON -> **Reveal the Fraud Ring**.
6. **Verify:** Click node -> **GraphRAG Explainer** shows the Deed with the undervalued price.

## 6. Success Metrics

1. **Response Time:** Graph rendering for the "Investigator View" (local neighborhood) must load in **< 2 seconds**.
2. **Clarity:** The "Before/After" ER toggle must be visually distinct enough to be understood without explanation.
3. **Integration:** The GraphRAG sidebar must correctly display the specific filename (e.g., `deed_102.txt`) alongside the graph node.
